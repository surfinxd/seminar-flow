# Architecture Decision Records

### ADR-005: GitOps 기반 배포 도구(ArgoCD) 채택
**결정**: Git 저장소를 단일 진실 공급원(Single Source of Truth)으로 사용하는 GitOps 구현체로 ArgoCD를 채택
**이유**: Git 상태와 쿠버네티스 클러스터 상태를 자동으로 동기화(Automated Sync)하여 `kubectl` 수동 조작에 따른 휴먼 에러를 방지하고, 배포 현황을 UI 기반으로 시각화하여 모니터링 편의성을 높이기 위함입니다.
**트레이드오프**: 초기 인프라 셋업을 위한 학습 및 설정 시간 비용이 발생하며, 파이프라인에서 Git 저장소를 갱신하는 봇(Bot)의 쓰기 권한 관리가 추가로 요구됩니다.

### ADR-006: 쿠버네티스 매니페스트 관리 도구(Kustomize) 채택
**결정**: Helm 대신 Kustomize를 사용하여 매니페스트 관리를 수행하고 환경(Dev/Prod)을 분리
**이유**: 공통 설정을 담은 `base` 디렉토리를 재사용하면서 `overlays` 디렉토리를 통해 환경별 설정(이미지 태그, 레플리카 수 등)을 직관적으로 덮어쓸(Patch) 수 있고, 새로운 문법 학습 없이 순수 YAML을 유지할 수 있어 유지보수성이 높기 때문입니다.
**트레이드오프**: 설정이 매우 복잡해지거나 조건부 로직(If/Else)이 강하게 필요한 경우에는 순수 YAML 패치 방식의 Kustomize보다 Helm이 더 적합할 수 있습니다.

### ADR-007: CI/CD 파이프라인(GitHub Actions) 연동
**결정**: 코드 통합, TDD 기반 테스트, 도커 이미지 빌드/푸시 및 Kustomize 태그 갱신 자동화 도구로 GitHub Actions 사용
**이유**: 소스코드가 관리되는 GitHub 저장소와 즉시 연동되며, 별도의 CI 서버 구축(Jenkins 등) 없이 관리형 서비스를 통해 손쉽고 일관성 있는 자동화 파이프라인을 구축할 수 있기 때문입니다.
**트레이드오프**: 플랫폼(GitHub)에 대한 강한 종속성이 생기며, 복잡한 파이프라인 구성 시 YAML 문법 디버깅이 다소 까다로울 수 있습니다.

### ADR-008: 컨테이너 레지스트리(Google Artifact Registry) 선정
**결정**: 빌드된 애플리케이션 도커 이미지의 중앙 저장소로 Google Artifact Registry(GAR) 채택
**이유**: 추후 도입될 GCP GKE 클러스터 인프라와의 연동이 가장 자연스럽고, GCP IAM 기반의 안전하고 철저한 보안 제어가 가능하기 때문입니다.
**트레이드오프**: 타 클라우드 프로바이더 환경이나 로컬 환경(k3d 등)에서 접근 시 별도의 GCP 자격 증명(Credentials) 설정과 관리가 수반되어야 합니다.

### ADR-009: 인프라 관리 도구(Terraform) 채택
**결정**: GCP의 VPC 네트워크, 서브넷, GKE 클러스터, Node Pool, Google Artifact Registry 및 IAM 권한 설정을 코드로 정의하고 관리(IaC)하기 위해 Terraform 채택
**이유**: 수동 인프라 구축에 따른 리소스 설정 오류를 방지하고 동일한 환경을 일관되게 복제할 수 있으며, 버전을 Git으로 추적하여 협업 및 복구 속도를 극대화하기 위함입니다.
**트레이드오프**: Terraform State 파일 관리(협업 시 원격 백엔드 필요) 및 인프라의 상태 변화에 대한 관리가 추가적으로 요구되며 사용법 학습이 선행되어야 합니다.

### ADR-010: Sealed Secrets를 활용한 선언적 보안 정보 관리
**결정**: Kubernetes 클러스터에 민감한 보안 정보(데이터베이스 비밀번호, JWT 비밀 키 등)를 GitOps 파이프라인에서 안전하게 관리하기 위해 Bitnami의 `Sealed Secrets`를 채택
**이유**: Git 저장소를 단일 진실 공급원(Single Source of Truth)으로 유지하는 GitOps 환경에서, 보안 비밀번호를 평문으로 노출시키지 않고 오직 클러스터 내부의 Sealed Secrets 컨트롤러만이 복호화할 수 있도록 비대칭 암호화를 적용함으로써, 보안성과 배포 자동화를 모두 달성하기 위함입니다.
**트레이드오프**: 보안 정보를 변경하거나 추가할 때마다 `kubeseal` CLI를 사용하여 암호화해야 하는 추가 절차가 발생하며, 클러스터 장애 복구 시 복호화용 마스터 비밀 키(보통 쿠버네티스 비밀 키)를 안전하게 백업 및 복구해야 하는 운영 관리 책임이 따릅니다.
